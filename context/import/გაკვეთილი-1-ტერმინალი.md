## Git-ის საფუძვლები (commit, push, branch)

Git არის ვერსიების კონტროლის სისტემა — ინახავს თქვენი პროექტის ისტორიას, ნაბიჯ-ნაბიჯ.

**მთავარი ცნებები:**

| ცნება | მნიშვნელობა |
|---|---|
| **repository (repo)** | folder, რომელსაც Git ადევნებს თვალს |
| **commit** | "სურათი" თქვენი ფაილების მდგომარეობისა, კონკრეტულ მომენტში, აღწერით |
| **branch** | პარალელური ვერსია პროექტისა (მაგ. ახალი feature-ის გასატესტად, ისე რომ მთავარი ვერსია არ დაზიანდეს) |
| **push** | ცვლილებების გაგზავნა ლოკალურიდან online-ზე (მაგ. GitHub) |
| **pull** | online-ზე არსებული ცვლილებების ჩამოტვირთვა ლოკალურად |

**ძირითადი ბრძანებები:**

```
git status          # რა შეიცვალა?
git add .            # ყველა ცვლილების მონიშვნა commit-ისთვის
git commit -m "..."  # ცვლილების შენახვა აღწერით
git push             # გაგზავნა GitHub-ზე
git pull             # ჩამოტვირთვა GitHub-დან
git branch           # branch-ების სია
git checkout -b ახალი-feature   # ახალი branch-ის შექმნა და გადასვლა
```

**Claude Code-თან ერთად:** არ არის საჭირო ამ ბრძანებების ზეპირად სწავლა — შეგიძლიათ უბრალოდ სთხოვოთ ბუნებრივ ენაზე, მაგ.: "შეინახე ეს ცვლილებები commit-ად" ან "გააკეთე push GitHub-ზე". Claude აირჩევს სწორ ბრძანებებს.

**გაფრთხილება:** `push` და `force push` შეიძლება შეუქცევადი იყოს გაზიარებულ პროექტებზე — ყოველთვის გადაამოწმეთ რას აკეთებს, სანამ დაადასტურებთ.

## Claude Code-ის agent-ების გამოყენება

**Agent-ი** არის Claude-ის ცალკე "ასისტენტი", რომელსაც მისცემთ კონკრეტულ დავალებას და თავად მუშაობს დამოუკიდებლად, საკუთარი ხელსაწყოებით.

**როდის გამოვიყენოთ:**
- დიდი, მრავალნაბიჯიანი დავალებები (მაგ. "გადაამოწმე მთელი კოდი შეცდომებზე")
- კვლევითი ამოცანები (მაგ. "მოძებნე ყველა ფაილი, სადაც გამოიყენება ეს ფუნქცია")
- ამოცანები, რომლებიც დამოუკიდებლად შეუძლია გაუშვას, სანამ თქვენ სხვა საქმეს აკეთებთ

**როგორ ვუშვებთ:**
- დააჭირეთ `←` ღილაკს — გაიხსნება agent-ების მენიუ
- ან უბრალოდ სთხოვეთ Claude-ს ბუნებრივ ენაზე, მაგ.: "გამოიყენე agent რომ მოძებნო X" — Claude თავად გადაწყვეტს საჭიროა თუ არა agent

**მნიშვნელოვანი:** agent-ი მუშაობს "სუფთა მეხსიერებით" — არ ახსოვს წინა საუბარი, ამიტომ დავალება უნდა იყოს მკაფიო და თვითკმარი.

## პროექტზე რეალური მუშაობის დაწყება (Ritualis ან TrottiLink)

როცა მზად იქნებით ნამდვილ პროექტზე მუშაობისთვის, აი რეკომენდებული თანმიმდევრობა:

1. **გახსენით სწორი folder** ტერმინალში — Ritualis-ის ან TrottiLink-ის repo, არა `jarvis-starter-kit` (თუმცა `/prime`-ით ჩატვირთული კონტექსტი მაინც გეხმარებათ, თუ Jarvis-ი იმავე parent folder-შია)
2. **დაიწყეთ მცირე** — პირველი დავალება იყოს კონკრეტული და მარტივი, მაგ.: "ახსენი რას აკეთებს ეს ფაილი" ან "დაამატე ერთი ღილაკი ამ ეკრანზე"
3. **გადაამოწმეთ შედეგი** — გაუშვით აპლიკაცია/ტესტები, სანამ commit გააკეთებთ
4. **commit ხშირად, მცირე ნაბიჯებით** — ერთი ლოგიკური ცვლილება = ერთი commit
5. **დიდი ცვლილებების წინ** — სთხოვეთ Claude-ს გეგმა (plan), სანამ კოდს დაწერს, განსაკუთრებით თუ ეხება არქიტექტურას ან მონაცემთა ბაზას

**Ritualis-სთვის კონკრეტულად:** ფოკუსი — App Store/Google Play-სთვის მზადება (UX/UI ოპტიმიზაცია, bug-ების გასწორება, store-ის მოთხოვნების დაკმაყოფილება).

**TrottiLink-სთვის კონკრეტულად:** ჯერ ისევ სტრატეგიული/დაგეგმვის ეტაპია — Claude-სთან მუშაობა შეიძლება დაიწყოს business model-ის დოკუმენტირებითა და MVP-ის feature-ების სიის შედგენით, სანამ კოდირებას დაიწყებთ.
